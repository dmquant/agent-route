Key Design Decisions

Why Unify Workflow + Session Execution?

The original design had workflows running through a separate code path from chat sessions, creating a "two-world" problem. Today's unification ensures a single execution model:

One pipeline, all paths

Whether triggered by chat WebSocket or workflow REST API, every agent execution flows through task_manager.create_task(). No special cases.

Brain Inspector everywhere

Workflow steps emit the same events as chat turns: AGENT_SELECTED, TOOL_CALL, TOOL_RESULT, METRIC. The inspector is automatically compatible.

Cross-triggering

Chat can run workflows in-session. Workflows can reuse existing sessions. The boundary between "interactive" and "batch" execution dissolves.

1-hour timeout for agentic tasks

LLM agents running complex multi-step tasks need time. The 120s default was causing premature timeouts. Upgraded to 3600s for real-world pipeline use.
